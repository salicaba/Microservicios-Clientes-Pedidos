1. Introducción

El presente documento describe el procedimiento completo para la implementación de un sistema basado en microservicios para la gestión de clientes y pedidos. El proyecto fue desarrollado como parte de la Actividad 3.1 de la materia Taller de Desarrollo 4, implementando una arquitectura hexagonal que separa claramente las capas de dominio, aplicación e infraestructura.

Los microservicios desarrollados son:

  • Microservicio de Clientes: Gestiona el CRUD completo de clientes utilizando MySQL como base de datos.
  • Microservicio de Pedidos: Gestiona el CRUD completo de pedidos utilizando PostgreSQL como base de datos.

La comunicación entre servicios se realiza a través de HTTP para validación de clientes y mediante RabbitMQ para mensajería asíncrona de eventos.

2. Objetivos

2.1 Objetivo General

Implementar un sistema de microservicios que simule la interacción entre servicios de clientes y pedidos, demostrando los principios de arquitectura orientada a servicios y comunicación asíncrona.

2.2 Objetivos Específicos

  • Desarrollar dos microservicios independientes con FastAPI.
  • Implementar CRUD completo (GET, POST, PUT, DELETE) en ambos servicios.
  • Utilizar bases de datos diferentes (MySQL y PostgreSQL).
  • Implementar comunicación HTTP entre servicios.
  • Configurar RabbitMQ para mensajería asíncrona.
  • Aplicar arquitectura hexagonal (Domain, Application, Infrastructure).
  • Contenerizar todos los servicios con Docker Compose.

3.2 Estructura Hexagonal

Cada microservicio sigue la arquitectura hexagonal con tres capas principales:

  • Domain: Contiene las entidades y repositorios abstractos.
  • Application: Implementa la lógica de negocio y servicios.
  • Infrastructure: Contiene la implementación concreta de bases de datos, API y mensajería.
microservicios-clientes-pedidos/ ├── clientes-service/ │ ├── domain/ # Entidades y repositorios │ ├── application/ # Lógica de negocio │ └── infrastructure/ # API, DB, RabbitMQ ├── pedidos-service/ │ ├── domain/ │ ├── application/ │ └── infrastructure/ ├── docker-compose.yml └── README.md

5. Endpoints de la API

5.1 Microservicio de Clientes (http://localhost:8001)

GET /clientes - Listar todos los clientes
POST /clientes - Crear un nuevo cliente
GET /clientes/{id} - Obtener un cliente por ID
PUT /clientes/{id} - Actualizar un cliente completo
DELETE /clientes/{id} - Eliminar un cliente

5.2 Microservicio de Pedidos (http://localhost:8002)

GET /pedidos - Listar todos los pedidos
POST /pedidos - Crear un nuevo pedido
GET /pedidos/{id} - Obtener un pedido por ID
PUT /pedidos/{id} - Actualizar un pedido completo
DELETE /pedidos/{id} - Eliminar un pedido
GET /clientes/{id}/pedidos - Listar pedidos de un cliente específico

6. Comunicación entre Servicios

6.1 Comunicación Síncrona (HTTP)

El microservicio de Pedidos realiza una petición HTTP al microservicio de Clientes para validar que el cliente existe antes de crear un pedido:

POST /pedidos → Validar cliente → GET /clientes/{id} → Si existe → Crear pedido

6.2 Comunicación Asíncrona (RabbitMQ)

Eventos publicados por Clientes y consumidos por Pedidos:

  • CLIENTE_CREADO: Cuando se crea un nuevo cliente
  • CLIENTE_ACTUALIZADO: Cuando se actualiza un cliente
  • CLIENTE_ELIMINADO: Cuando se elimina un cliente

10. Conclusiones

El desarrollo de este proyecto permitió:

  1. Implementar exitosamente dos microservicios independientes con FastAPI.
  2. Aplicar arquitectura hexagonal separando claramente las capas de dominio, aplicación e infraestructura.
  3. Configurar dos bases de datos diferentes (MySQL y PostgreSQL) para cada microservicio.
  4. Establecer comunicación HTTP entre servicios para validación de datos.
  5. Implementar mensajería asíncrona con RabbitMQ para eventos.
  6. Contenerizar todos los servicios con Docker Compose.
  7. Documentar completamente el proyecto con Swagger.

Lecciones Aprendidas

  • La arquitectura hexagonal facilita el mantenimiento y la escalabilidad.
  • Docker Compose simplifica la orquestación de múltiples servicios.
  • La comunicación asíncrona mediante RabbitMQ desacopla los servicios.
  • La validación de datos entre servicios mejora la integridad del sistema.
